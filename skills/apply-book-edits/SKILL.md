# Apply book edits from a highlights export

The author reads the book on the site, highlights passages, and writes a **note** on
each one describing the change they want. They export those highlights to a JSON file
and hand it to you. Your job is to turn each note into an edit to the book's text.

## What the input looks like

An export is either a wrapper object or a bare array of highlight records:

```json
{
  "app": "cpds", "type": "highlights", "version": 1,
  "book": "Christian Preaching That Devalues Scripture",
  "exportedAt": "2026-06-06T12:00:00.000Z",
  "highlights": [
    { "id": "h…", "chap": 1, "cidx": 0, "start": 27, "end": 41,
      "text": "1Kings 4:29-34", "note": "Spell out '1 Kings' with a space.", "ts": 1 }
  ]
}
```

Field meaning:

- **chap** — chapter number. `0` is the **Dedication** (it lives on `contents.html`).
- **cidx** — zero-based paragraph index within that chapter (`<p data-cidx="N">`).
- **start / end** — character offsets of the highlight within the paragraph's plain text.
- **text** — the exact highlighted text (plain, with entities already decoded).
- **note** — the author's instruction. **A highlight with an empty note is not an edit
  request** — it's just a reading highlight. Skip it unless told otherwise.

## Where the text lives (read this — it's the whole trick)

The site is generated by `build_site.py` from the PDF, but **you do not edit the PDF and
you do not rebuild.** You edit the generated HTML in `site/`. Each book paragraph appears
in **two** files that must be kept identical:

1. **Scroll view** — `site/chapter-<chap>.html`, inside `<p data-cidx="<cidx>">…</p>`.
   Text here is **HTML-escaped**: `&`→`&amp;`, `<`→`&lt;`, `>`→`&gt;`. Quotes and the
   book's curly quotes (`“ ” ‘ ’`) are literal UTF-8. (Chapter `0` is in
   `site/contents.html`, in the Dedication's `<div class="prose" data-chap="0">`.)
2. **Flip view** — `site/book.html`, inside the `#bookdata` JSON, at
   `BOOK[chap-1].paras[cidx]`. Text here is a **raw JSON string** (escape `"` and `\`).
   The Dedication is **not** in the flip view, so chapter `0` has no flip target.

If you change only one, the two views disagree. Always change both (except chap 0).

Do **not** worry about the `<a class="scripture">` links — those are added in the browser
at runtime, so the stored HTML paragraphs are plain text. Editing prose won't break them.

## Workflow (propose first, then apply)

### 1. Locate the file and load it
If the user gave a path, use it. Otherwise ask, or look in the working folder / uploads
for a `*.json` highlights export. Note the path to `build_site.py` to confirm you're at
the repo root; the site is under `site/`.

### 2. Build the edit plan with the resolver (read-only)
Run the helper, which maps every noted highlight to its exact locations and flags any
problems (stale offsets, out-of-sync views, missing paragraphs):

```bash
python3 skills/apply-book-edits/resolve_highlights.py <HIGHLIGHTS.json> --site site
```

Add `--json` for a machine-readable plan, `--all` to include note-less highlights.
Read the warnings (`!!`). If offsets no longer match, locate the passage by its **text**,
not its offsets — the paragraph was probably edited already.

### 3. Interpret each note and draft the change
Notes are natural language. Read the note together with the highlighted text **in its
paragraph context** (the resolver prints it) and decide the minimal edit that satisfies
the intent. Typical notes and how to treat them:

- *"fix typo: X → Y"*, *"change A to B"* — literal replacement.
- *"reword / tighten / clarify this"* — rewrite the highlighted span (or the sentence it
  sits in) while preserving meaning, tense, and the author's voice. Change as little as
  the note requires.
- *"delete this" / "cut"* — remove the span and repair surrounding spacing/punctuation.
- *"add … here" / "insert …"* — add text at the highlighted point.
- *"add a comma" / formatting* — apply within the paragraph text.
- Notes that ask for **structural** changes (split/merge/move paragraphs, renumber
  chapters) are bigger than a paragraph edit — surface them to the author and handle
  separately rather than forcing them into one paragraph.

When the intent is ambiguous, ask the author rather than guessing.

### 4. Propose, then wait for approval
Present a concise, numbered proposal — one row per noted highlight — showing chapter/
paragraph, the note, and a **before → after** of the affected text. Group obviously
mechanical fixes; spell out judgment-call rewrites in full. Do not edit any file until
the author approves. (If the author has pre-approved "apply directly," you may skip the
wait, but still produce the summary.)

### 5. Apply each approved change to BOTH views
For each approved edit:

- **Scroll view**: edit the text inside `<p data-cidx="<cidx>">` in
  `site/chapter-<chap>.html` (or `contents.html` for chap 0). Keep the `data-cidx`
  attribute. HTML-escape `& < >` in any new text; leave curly quotes as-is.
- **Flip view** (chap ≥ 1): edit `BOOK[chap-1].paras[cidx]` inside the `#bookdata`
  JSON in `site/book.html`. Keep it valid JSON (escape `"` and `\`).

Make the final text in both places **identical** (modulo each file's escaping).

### 6. Record every change in the changelog
Append an entry to `EDITS_CHANGELOG.md` at the repo root (create it from the template
there if missing) for each applied edit: date, highlight id, chapter/paragraph, the note,
and the before/after text. This is the durable record of what changed — see the rebuild
warning below for why it matters.

### 7. Verify
- Re-run the resolver with `--verify`. For passages whose text changed, expect
  `scroll=False flip=False` (the original text is gone). For non-text notes (e.g. "add a
  margin comment") it may still be present — that's fine.
- Confirm `site/book.html` still parses: its `#bookdata` must remain valid JSON
  (`python3 -c "import json,re; d=open('site/book.html').read(); json.loads(re.search(r'id=.bookdata.>(.*?)</script>', d, 16).group(1)); print('bookdata OK')"`).
- Spot-check that scroll and flip text now match for each edited paragraph.

### 8. Report
Summarize what changed and what was skipped (note-less highlights, structural requests,
anything needing author input).

## ⚠ Critical: do not rebuild from the PDF after editing
`build_site.py` regenerates every HTML file from `Book After Xulon Version.pdf` and will
**overwrite these edits**. After applying author edits, do not run `build_site.py` for a
text change. If a rebuild is ever required (e.g. to change site features, CSS, or JS),
the `EDITS_CHANGELOG.md` is the source of truth: re-apply each logged change to the freshly
generated HTML afterward. Treat the changelog as authoritative for the book's true text.
