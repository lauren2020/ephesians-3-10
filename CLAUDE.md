# CLAUDE.md — context for AI agents working in this repo

This repository builds and hosts a single book as a minimalist reading website:
**“Christian Preaching That Devalues Scripture.”** It's a static site deployed on
Netlify (no build step on deploy — the pre-built `site/` folder is published directly).

## Repository map

| Path | What it is |
|------|------------|
| `Book After Xulon Version.pdf` | The original manuscript. Source for the **initial** text extraction. |
| `build_site.py` | Generator. Extracts text from the PDF with `pdftotext` and writes all of `site/`. |
| `scripture_lib.py` | Helper used by the build to detect & resolve KJV scripture references. |
| `site/` | The built, deployed website (HTML/CSS/JS). **This is what's served.** |
| `site/chapter-N.html` | Scroll-view reading pages (one per chapter). |
| `site/contents.html` | Table of contents + the Dedication. |
| `site/book.html` | Flip-book view. Holds its own copy of the text in a `#bookdata` JSON blob. |
| `site/index.html` | Cover page. |
| `site/scripture.js` | Runtime scripture-reference linkifier + passage modal. |
| `site/highlights.js` | Reader highlighting: select-to-highlight, notes, panel, export/import. |
| `site/style.css` | All styles (hand-maintained; **not** generated by `build_site.py`). |
| `netlify.toml` | Publishes `site/` with no build command. |
| `skills/apply-book-edits/` | Skill + helper for applying author edits from a highlights export. |
| `EDITS_CHANGELOG.md` | Durable record of every manual edit made to the book text. |

## Building the site

```bash
pip install pythonbible --break-system-packages   # build dependency
# pdftotext (poppler-utils) must be installed
python3 build_site.py                              # regenerates everything in site/
```

`build_site.py` is the source of truth for the site's **structure, features, and styling
hooks** — paragraph anchors, scripture data, the flip view, etc. `style.css` is edited by
hand.

## ⚠ The most important thing to know: text edits vs. rebuilds

`build_site.py` **regenerates every file in `site/` from the PDF.** It does not read the
HTML back in. Therefore:

- **Any edit made directly to `site/*.html` is erased the next time the build runs.**
- The book's text is edited **in the generated HTML**, per the author's chosen workflow,
  **not** in the PDF.
- Because of that, **`EDITS_CHANGELOG.md` is the authoritative record of the book's true
  text.** Every manual text change must be logged there.
- **Do not run `build_site.py` to make a text change.** Only rebuild for structural/
  feature/CSS/JS work — and when you do, re-apply the changes in `EDITS_CHANGELOG.md` to
  the freshly generated HTML afterward.

A book paragraph appears in **two** places that must stay in sync:

1. `site/chapter-<chap>.html` → `<p data-cidx="<cidx>">…</p>` (HTML-escaped text).
   Chapter `0` (the Dedication) is in `site/contents.html`.
2. `site/book.html` → `#bookdata` JSON at `BOOK[chap-1].paras[cidx]` (raw JSON string).
   The Dedication is not in the flip view.

Note: `<a class="scripture">` links are injected **in the browser at runtime**, so the
stored HTML paragraphs are plain text. Editing prose does not break scripture links.

## The reader highlight system (and its export format)

`site/highlights.js` lets readers select text, save highlights (in `localStorage` under
`cpds:highlights`), attach notes, view them in a slide-out panel, and **export/import**
them as JSON. A highlight record:

```json
{ "id": "h…", "chap": 1, "cidx": 0, "start": 27, "end": 41,
  "text": "the highlighted text", "note": "author's instruction", "ts": 1700000000000 }
```

`chap`/`cidx` identify the paragraph (matching the `data-chap`/`data-cidx` attributes in
the HTML); `start`/`end` are character offsets within that paragraph's plain text.

## Applying author edits from a highlights export

The author communicates desired revisions by highlighting passages, writing a **note** on
each describing the change, and exporting the highlights to JSON. To apply them, use the
**`apply-book-edits`** skill (`skills/apply-book-edits/SKILL.md`). In short:

1. `python3 skills/apply-book-edits/resolve_highlights.py <export.json> --site site`
   resolves every noted highlight to its exact edit locations and flags problems.
2. Propose each change (before → after) and get the author's approval.
3. Apply each approved edit to **both** the scroll-view file and the flip-view JSON.
4. Log every change in `EDITS_CHANGELOG.md`.
5. Verify with `--verify` and confirm `#bookdata` is still valid JSON.

Highlights **without a note are not edit requests** — skip them.

> The skill lives in `skills/` so it's tracked with the repo. To have Claude Code load it
> automatically, copy or symlink `skills/apply-book-edits` into `.claude/skills/`
> (e.g. `mkdir -p .claude/skills && ln -s ../../skills/apply-book-edits .claude/skills/`).

## Conventions

- Preserve `data-chap` / `data-cidx` attributes — the highlight system anchors to them.
- Keep scroll-view and flip-view text identical for any edited paragraph.
- HTML-escape `& < >` in HTML files; keep the book's curly quotes (`“ ” ‘ ’`) as literal
  characters. Keep `#bookdata` valid JSON.
- Match the author's voice and change as little as each note requires.
- Don't reformat or reflow files you aren't editing.
