# CLAUDE.md — context for AI agents working in this repo

This repository builds and hosts a single book as a minimalist reading website:
**“Christian Preaching That Devalues Scripture.”** It's a static site deployed on
Netlify (no build step on deploy — the pre-built `site/` folder is published directly).

## Repository map

| Path | What it is |
|------|------------|
| `manuscript/` | **The book's text — the editable source of truth.** One file per chapter. |
| `manuscript/ch01.txt … ch13.txt` | Chapter text. Paragraphs separated by a blank line. |
| `manuscript/dedication.txt` | The Dedication (chapter `0`). |
| `build_site.py` | Generator. Reads `manuscript/` and writes all of `site/`. |
| `scripture_lib.py` | Helper used by the build to detect & resolve KJV scripture references. |
| `Book After Xulon Version.pdf` | Original manuscript. **Historical artifact only** — used to seed the text once; no longer the build source. |
| `site/` | The built, deployed website (HTML/CSS/JS). **This is what's served.** Generated — don't hand-edit. |
| `site/chapter-N.html` | Scroll-view reading pages (one per chapter). |
| `site/contents.html` | Table of contents + the Dedication. |
| `site/book.html` | Flip-book view. Holds its own copy of the text in a `#bookdata` JSON blob. |
| `site/index.html` | Cover page. |
| `site/scripture.js` | Runtime scripture-reference linkifier + passage modal. |
| `site/highlights.js` | Reader highlighting: select-to-highlight, notes, panel, export/import. |
| `site/search.js` | Header search box: lazy-loads the index, matches, jumps to a paragraph. |
| `site/search-index.js` | Generated search index (all paragraphs, keyed by chapter + cidx); sets `window.__CPDS_INDEX__`, lazy-loaded by `search.js` on first search. |
| `site/style.css` | All styles (hand-maintained; **not** generated by `build_site.py`). |
| `netlify.toml` | Publishes `site/` with no build command. |
| `skills/apply-book-edits/` | Skill + helper for applying author edits from a highlights export. |
| `EDITS_CHANGELOG.md` | Human-readable history of book-text edits. |

## How the text and the site relate

`manuscript/*.txt` is the **single source of truth for the book's text.** `build_site.py`
reads it and regenerates everything in `site/` — both the scroll view (`chapter-N.html`,
`contents.html`) and the flip view (`book.html`'s `#bookdata`). One paragraph edited in
the manuscript propagates to both views on rebuild. **Never hand-edit the book text in
`site/`** — it's generated and will be overwritten on the next build.

Manuscript format: paragraphs are separated by one or more blank lines; lines within a
paragraph are joined with a space. Keep the book's curly quotes (`“ ” ‘ ’`) as literal
characters; write `& < >` as themselves (the build HTML-escapes them). Scripture links
are added in the browser at runtime, so the manuscript is plain prose — editing it never
breaks scripture references.

> `build_site.py` falls back to extracting a chapter from the PDF only if that chapter's
> manuscript file is missing. With the manuscript present, the PDF and `pdftotext` are not
> needed to build.

## Building the site

```bash
pip install pythonbible pythonbible-kjv --break-system-packages   # build dependencies
python3 build_site.py                                              # regenerates site/
```

`pythonbible` detects scripture references; **`pythonbible-kjv` supplies the KJV verse
text** for the scripture pop-up modals. Without the KJV package, `pythonbible` returns
empty verse strings — so the build now aborts with an error rather than shipping a modal
with blank verses. If you see that error, install `pythonbible-kjv` and rebuild.

The build prints each chapter's source (`manuscript/chNN.txt`). `style.css` is the only
part of `site/` that is hand-maintained rather than generated.

## Editing the book text

1. Edit the relevant `manuscript/*.txt` file (chapter `N` → `chNN.txt`; Dedication →
   `dedication.txt`). Edit only the paragraphs you mean to; don't reflow the rest.
2. Run `python3 build_site.py` to regenerate `site/`.
3. Log the change in `EDITS_CHANGELOG.md`.

Rebuilding is safe and expected — it's how edits reach the site.

## Applying author edits from a highlights export

Readers (the author) can highlight passages, attach a **note** describing a desired
change, and export their highlights to JSON (see the highlight system below). To turn
those notes into edits, use the **`apply-book-edits`** skill
(`skills/apply-book-edits/SKILL.md`). In short:

1. `python3 skills/apply-book-edits/resolve_highlights.py <export.json>` resolves every
   noted highlight to its manuscript file + paragraph and flags problems.
2. Propose each change (before → after) and get the author's approval.
3. Apply each approved edit to the manuscript paragraph.
4. Log it in `EDITS_CHANGELOG.md`, rebuild, and verify with `--verify`.

Highlights **without a note are not edit requests** — skip them.

> The skill lives in `skills/` so it's tracked with the repo. To have Claude Code load it
> automatically, copy or symlink `skills/apply-book-edits` into `.claude/skills/`
> (e.g. `mkdir -p .claude/skills && ln -s ../../skills/apply-book-edits .claude/skills/`).

## The reader highlight system (and its export format)

`site/highlights.js` lets readers select text, save highlights, attach notes, organize
them into named **groups** (switch the active group, create/rename/delete groups), view
them in a slide-out panel, and **export/import** as JSON. Highlights are stored in
`localStorage` under `cpds:hlgroups` as `{v, activeId, groups:[{id, name, highlights:[…]}]}`
(a legacy flat `cpds:highlights` array is migrated into a default group on first load).
A highlight record:

```json
{ "id": "h…", "chap": 1, "cidx": 0, "start": 27, "end": 41,
  "text": "the highlighted text", "note": "author's instruction", "ts": 1700000000000 }
```

An export wraps one group's highlights:

```json
{ "app": "cpds", "type": "highlights", "version": 1, "book": "…", "group": "Sermon prep",
  "exportedAt": "…", "highlights": [ { …record… } ] }
```

Each export/import operates on a single group; an import lands in the active group, or in a
new group when "Import as a new group" is chosen.

`chap`/`cidx` identify the paragraph (matching the `data-chap`/`data-cidx` attributes in
the generated HTML, which in turn match the manuscript's chapter file and paragraph
order); `start`/`end` are character offsets within that paragraph's text.

## Conventions

- The manuscript is the source of truth for text; `site/` is generated (except `style.css`).
- Preserve paragraph order in manuscript files — `cidx` anchors (and reader highlights)
  depend on it. Adding/removing a paragraph shifts the indices after it.
- Keep curly quotes literal; let the build handle HTML/JSON escaping.
- Match the author's voice and change as little as each edit requires.
- Don't reformat or reflow files you aren't editing.
